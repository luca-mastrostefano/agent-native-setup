# `profile save`: extract a reusable profile from a project's delta

- **Status:** Proposed
- **Date:** 2026-07-03
- **Author:** Luca Mastrostefano
- [ ] Implemented

## Context

Authoring a profile *from scratch* is well served today: `profile init` (skeleton + an authoring
`AGENTS.md`), build up `templates/`, `profile validate`. But a common path has no tool support:
*"I scaffolded a project, customized it — added `.claude/` agents, edited the contract, added
house rules — and want to reuse **that** as a profile for new projects."*

Doing it by hand is exactly the fiddly, error-prone job the `/tmp` experiment earlier in this
project exposed: copying the whole scaffold produces a **frozen, lossy photocopy** — the project
name baked in, the write-once/`seed` vs. managed distinction lost (so a profile update would
overwrite the user's README), symlinks (`CLAUDE.md`/`GEMINI.md`) shipped as broken duplicate
files, and unrelated scratch swept in. Stage A of RFC 2026-07-03 (ecosystem-core) names
`profile save` as an unbuilt prerequisite; this RFC pins its design.

The enabling asset: a scaffolded project carries a `.agent-native-setup.json` manifest — the
recorded config, a **fingerprint per generated file**, and the `seed` set. `update` already uses
those fingerprints to classify each file as pristine / edited / user-added. `profile save` is the
*inverse* of that classification: instead of refreshing what's pristine, it **captures what
diverged.**

## Decision

`agent-native-setup profile save <project> -o <profile-dir> [--name NAME]` produces an
`extends: default` profile containing the project's **delta from the default** — reusing the same
regenerate-and-compare machinery as `update`, never touching the source project (read-only).

1. **Read the manifest** (`<project>/.agent-native-setup.json`); error clearly if absent ("run it
   on a project agent-native-setup scaffolded").
2. **Regenerate the recorded default** (from the manifest's `config`) into a temp dir — the
   baseline the project was born from. (This is what `update._regenerate` already does.)
3. **Classify each of the project's files** against that baseline + the manifest fingerprints, and
   keep only the delta:
   - **edited managed file** (on-disk ≠ baseline) → include as a **managed** template (the user's
     version) — the default's copy is superseded by the profile.
   - **user-added file** (present, not generated by the baseline) → include as **managed**. This
     is a *decision*, not a default-to-revisit: it's house content the author is choosing to ship
     and will maintain, so a profile update *should* propagate it — and a consumer's later edit
     becomes a **reported conflict** (never a silent clobber) via `update`'s existing
     classification, which is exactly what makes "managed" safe here. (A future per-file override
     could mark specific added files `seed`; managed is the correct default.)
   - **pristine managed file** (on-disk == baseline) → **exclude** — the default already provides
     it, and the profile `extends` it.
   - **edited seed file** (e.g. a customized `README`/architecture overview) → include as **`seed`**
     (write-once), so it seeds a starter but a consumer still owns it. Pristine seed files are the
     default's and are excluded.
4. **Parameterize the project identity — conservatively.** The manifest records `project_name`;
   `slug` is *derived* (never stored), so `save` recomputes it. In captured text files, substitute
   only **word-boundary** matches of the name/slug (never a substring inside a larger word, a URL,
   or a package name) with `{{ project_name }}` / `{{ slug }}`, renaming touched files to `.j2`.
   Ambiguous or substring occurrences are **left literal** and listed in the report, *not* silently
   rewritten — a blanket replace that corrupts a file where the name appears as a substring is
   worse than an under-parameterized draft the author finishes by hand. `save` reports every
   substitution **and** every skipped candidate, and tells the author to review. (A tighter future
   variant — substitute only at the exact sites the generators actually rendered the name — needs
   generate-time provenance the manifest doesn't record yet; word-boundary is the pragmatic v1.)
5. **Symlinks can't be templated.** A captured symlink (e.g. `CLAUDE.md`→`AGENTS.md`) becomes an
   **`onboarding` step** to recreate it (`ln -s AGENTS.md CLAUDE.md`) rather than a broken
   duplicate file — reusing the authoring-harness insight that onboarding does what templates
   can't.
6. **Write the profile**: `profile.json` (`extends: default`, `version: 0.1.0`, `seed` = the
   captured seed files, `name` from `--name` or the dir name, a TODO description), the captured
   files under `templates/`, and the authoring `AGENTS.md`/`README.md` (as `profile init` writes).
   Exclude the manifest itself and transient files (`ONBOARDING.md`, `/onboard`).
7. **Classify and disclose its safety tier.** A saved profile almost always **carries code** by
   the ecosystem-core rule (RFC 2026-07-03-ecosystem-core §3–4): a captured `.github/workflows/*`
   or `Taskfile`, any `session_start` hooks, and the `ln -s` onboarding step from step 5 are all
   `unsafe`-tier. `save` runs that *same derived classification* on its own output, records the
   tier in the generated `profile.json`/report, and **tells the author their profile is
   code-carrying** — so they know a consumer will need `--allow-code` once that gate lands. `save`
   must never emit a profile it silently mislabels safe; when the classifier isn't yet built, it
   conservatively discloses "assume code-carrying."
8. **Summarize and hand off**: print what was captured (managed / seed counts), every name
   substitution *and skipped candidate*, every symlink turned into an onboarding step, and the
   safety tier — then tell the author to run `profile validate` and review before publishing.
   `save` produces a *draft to review*, not a finished artifact.

**Why delta, not snapshot.** A snapshot (the whole default copied into `templates/`) is the frozen
photocopy the `/tmp` run produced: name baked in, `seed` lost, no *living* logic (it can't
re-detect languages or pick up new pins). The delta yields a real `extends: default` profile that
**inherits the default's evolving logic** and carries only the team's actual changes — which is
what makes it reusable and updatable rather than a fossil.

## Consequences

**Easier / newly possible**
- The "customize a project, reuse it as a profile" path becomes one command — the main authoring
  gap for non-experts.
- **No new comparison logic**: `save` reuses `update`'s regenerate + fingerprint classification,
  so the *meaning* of "pristine vs edited" can't drift between the two. (The inverse direction does
  have a false-positive mode `update` never faces — a non-deterministically regenerated file; see
  Honest limits.)
- The captured profile is a *review-ready draft* that already `validate`s (harness + `AGENTS.md`
  included), not a raw dump.

**Harder / honest limits**
- **Name parameterization is conservative, not complete.** Word-boundary substitution avoids
  substring corruption but *under*-parameterizes: a name a generator transformed (or that appears
  mid-token) is left literal for the author to finish. `save` biases toward a safe, reviewable
  under-parameterized draft over a dangerous over-eager one. Multi-word names and `slug`/name
  collisions still need author review.
- **The regenerated baseline may be non-deterministic.** `save`'s delta assumes "regenerate the
  recorded config → byte-identical baseline," but the scaffold has known non-deterministic outputs
  (the date-stamped bootstrap RFC path, version strings) that `update` already special-cases. A
  pristine-but-non-identically-regenerated file would classify as *edited* and be wrongly captured
  as a delta — a **false-positive** the forward `update` direction never faces. Mitigation: exclude
  the same known non-deterministic paths `update` does, and the substitution/capture report lets
  the author drop a stray. Worth stating plainly rather than resting on "no new comparison logic."
- **Symlinks are captured at lower fidelity.** Templates can't carry a symlink, so `CLAUDE.md`→
  `AGENTS.md` becomes an `ln -s` *onboarding step* — meaning the link exists only **after** an
  agent runs onboarding, where the original project had it immediately. A consumer who inspects
  the fresh tree (or never runs the onboarding agent) sees no `CLAUDE.md`. Defensible (no
  alternative), but a real fidelity loss, not just a clever reuse.
- **Only works on a scaffolded project** (needs the manifest). Extracting from an arbitrary,
  never-scaffolded repo is out of scope — that's the messy diff-against-nothing problem, and the
  manifest is precisely what makes the clean version possible.
- **Binary/opaque user-added files** are captured verbatim (no parameterization); fine, but the
  author should confirm they belong in a profile.

**What we give up**
- Nothing that exists — `save` is additive (a new subcommand, read-only on the source). Fully
  reversible: it writes a new directory and can be deleted.

## Alternatives considered

- **Snapshot the whole scaffold** (copy every generated file into `templates/`). Rejected: the
  `/tmp` experiment proved it lossy — a frozen photocopy that loses `seed` status, bakes in the
  name, drops symlinks, and can't inherit the default's evolving logic. The delta is strictly
  better for a profile that `extends: default`.
- **Capture the manifest's `files` list wholesale** (no diff against a regenerated baseline).
  Rejected: that's the whole default, i.e. the snapshot again — the diff is what isolates the
  *delta*.
- **Interactive per-file selection** ("include this file? seed or managed?"). Deferred: start with
  automatic classification that's *reported and reviewable*; add interactivity only if the
  heuristic proves too blunt in practice. Simpler first.
- **Emit a standalone (`extends: null`) profile.** Rejected as the default: standalone re-freezes
  the whole setup. `extends: default` + delta is the reusable, updatable shape. (A `--standalone`
  variant could come later if someone genuinely wants the fossil.)

## Open questions (for implementation)

- **`slug` vs `project_name` token choice when they're equal.** Word-boundary matching (Decision
  step 4) removes the substring-corruption risk, but when `project_name` slugifies to itself, a
  matched occurrence could be emitted as either `{{ project_name }}` or `{{ slug }}`. Rule of
  thumb: use `{{ slug }}` where the file is a package/identifier context (`pyproject.toml` name,
  directory-ish uses) and `{{ project_name }}` for prose — but the exact call needs pinning, and
  the substitution report must show which token each site got.
- **A per-file `seed` override** so an author can promote a specific *user-added* file to
  write-once (the Decision makes user-added files managed by default; this is the escape hatch).
- **Scope of parameterization** — only the project name/slug, or also the description? Description
  is lower-risk to leave literal (the author edits it anyway).
