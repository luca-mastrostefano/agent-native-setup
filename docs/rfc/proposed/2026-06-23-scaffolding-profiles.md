# Scaffolding profiles: community-authored, composable, updatable setups

- **Status:** Proposed
- **Date:** 2026-06-23
- **Author:** Luca Mastrostefano
- [ ] Implemented

## Context

The wizard has exactly one generator: ours. It scaffolds *our* opinion of an
agent-native setup (the four principles, the `.claude/` library, our linters and CI).
That opinion is good, but it is one opinion. A team with a mature, distinctive setup —
Tolaria's, say (CodeScene health gates, a Codacy security gate, a release-readiness
completion checklist, shadcn rules, their own `.claude/` agents and MCP servers) — has no
way to say *"scaffold new projects exactly like ours."* Their only options are to fork the
tool or to copy a template repo by hand and re-diverge every time.

Two assets we've already built make a community profile ecosystem unusually cheap, and the
design hangs off both:

1. **The update engine doesn't know who generated the files.** `update.py` operates on a
   manifest contract — `{version, config, files: fingerprints, seed}` — and a regenerate +
   classify loop (create / refresh-pristine / conflict / orphan). It never references the
   wizard except through `cli.build`, which is the *one* hardcoded generator. The expensive,
   reusable half of "shareable setups that stay updatable" is done.
2. **The contract already splits method from project-specifics.** RFC 2026-06-20 split
   `AGENTS.md` (thin, user-owned) from `INSTRUCTION.md` (the managed standard method). A
   profile is the natural home for the *next* layer out: a team's house rules, layered on
   our base the way `INSTRUCTION.md` layers under `AGENTS.md`.

So the goal: let anyone author a **profile** — a packaged, versioned generator — that
*composes on our base* (or starts from blank), ships whatever content it needs (extra
agents, MCP config, startup hooks, house gates), and **inherits the update machinery** so a
project scaffolded from it stays refreshable. Profiles live either in a reviewed
`contributions/` directory we ship, or are distributed privately within a team.

Four constraints shape the design:

1. **Updatability needs the generator back at update time.** Our refresh-pristine logic is
   "regenerate from the saved config, compare to the fingerprint." That only works if the
   profile that produced a file can be **re-resolved and re-run** months later, at its
   latest version. A profile the engine can't find again can't refresh anything it wrote.
2. **Updatability needs determinism.** "Regenerate → byte-identical → safe to refresh" is
   the whole basis of classification. Same `(profile@version, config)` must produce
   identical bytes. (We've been bitten by nondeterminism before — hatch-vcs versions,
   `Date.now`.)
3. **Composition has two version axes, not one.** Today there is one version (ours). A
   composed project is generated by a *stack* of layers (base + child), each with its own
   release cadence. The update gate, the breaking-series rule, and the migration span
   (RFC 2026-06-22) must run **per layer**.
4. **A profile is executable trust.** Even a pure-data profile ships content that *runs* —
   SessionStart hooks, MCP servers, agents. Scaffolding from a third-party profile is "I
   trust this author's setup," the same trust as running any template repo. A profile that
   ships *code* (a generator entrypoint) is arbitrary execution at scaffold **and** update
   time. The community channel and the trust model have to account for that.

## Decision

Reframe the product around one interface: **the wizard becomes the built-in `default`
profile, and the engine resolves, runs, and updates a stack of profiles.** Ship it in two
phases so the high-value, low-risk core lands first. Seven coupled decisions.

### 1. A profile is a versioned, resolvable generator

A profile is a directory (zippable, git-hostable, or pip-installable):

```
my-profile/
  profile.json      # metadata + contracts
  templates/        # the files it ships, rendered against the resolved config
```

`profile.json`:

```json
{
  "name": "tolaria",
  "version": "1.2.0",          // the profile's OWN semver (axis 2)
  "extends": "default",        // "default" | null (null/absent = start from blank)
  "description": "Tolaria's agent setup: health gates, completion checklist, MCP servers",
  "update_source": "git+https://…",  // where `update --check` looks for newer versions (§7)
  "config_schema": { ... },    // the answers this profile accepts (or none = fully fixed)
  "seed": ["README.md", ...],  // this layer's seed-vs-managed taxonomy
  "migrations": [ ... ]        // this layer's version-keyed structural moves (Phase 2)
}
```

Because the profile ships its templates as plain files, **the MCP config, the startup
hooks, and the extra agents are just content in `templates/`** — they need no special
wizard feature. "Personalize the MCPs / startup agents" falls out of the file tree.

Profiles resolve through layered sources, first hit wins: an explicit `--profile <path>` →
the user dir `~/.config/agent-native-setup/profiles/<name>/` → the shipped
`contributions/<name>/` → the built-in `default`. The manifest records **how to
re-resolve** the profile (its `source`), which is what makes Constraint 1 satisfiable.

**No compile step.** A declarative profile is data: author a directory, drop it in
`contributions/` (PR) or your user dir, point `--profile` at it. Format is **JSON** to
start — zero new dependencies and the exact manifest schema, so profile and manifest stay
in sync by construction. A richer templating language (Jinja) and code profiles are
deferred (see §6, Alternatives).

### 2. `extends: default | null` — composition as a layer stack

A project is generated by an ordered **stack** of layers, base-first:

- **`extends: "default"`** → the stack is `[default, <profile>]`. The engine renders the
  base, then the child **adds** files at new paths and **overrides** files at shared paths.
  Each output path is owned by exactly one layer: the **topmost layer that produced it**.
- **`extends: null`** (or absent) → the stack is `[<profile>]`. Standalone: the profile is
  the sole generator, shipping none of our base content — but it *still* gets the manifest,
  the Scaffolder, and the update machinery, so even a from-blank profile is updatable.

The single-owner rule is what keeps update tractable: on refresh, each path is
re-resolved through *its* owning layer at *that* layer's latest version. Base improvements
flow to paths the base still owns; they do **not** flow to a path the child overrode (the
child claimed it — that's the point of an override).

### 3. Phase 1 — composition by seeding, reusing the classification engine

The MVP makes the child layer's files **`seed`** (write-once, user-owned). This needs *no*
new versioning and *no* per-child migrations — but it is **not** "zero engine changes." It
reuses the entire **classification / gate / migration** engine (classify, the seed
short-circuit, `decide`, `steps_in_span`) untouched; the one new piece is **stack
composition itself**, and that piece is small but real:

- The base (`default`) stays `managed` and fully updatable — base fixes still propagate.
- The child's **added** files (new paths) are `seed`: scaffolded once, then the user's to
  maintain. This is the genuinely free case — `Scaffolder.write` already writes a
  not-yet-existing path, and `classify` already short-circuits a `seed` path (never
  refreshed, never an orphan).
- A child **override** of a base-managed path is the case that costs code, and the RFC is
  explicit about it rather than hand-waving it onto the seed short-circuit:
  - **At scaffold time**, `Scaffolder.write` is *first-writer-wins* (`scaffold.py` skips a
    path that already exists), so a base-first stack would silently keep the base file and
    drop the child's. Composition must render the stack **child-last with intra-stack
    supersede** — a small, contained writer change (a layer may overwrite an earlier layer's
    not-yet-user-touched output), distinct from `--force` (which overwrites the *user's*
    files).
  - **At update time**, `classify`'s short-circuit fires on the *freshly regenerated* seed
    set, but base regeneration marks that path `managed`. So composition must **re-mark every
    child-overridden path into the regenerated seed set** before classify runs, or the base
    content refreshes back over the override. One bounded step in the compose layer — not a
    behavior the seed policy gives us for free.

So Phase 1 = the classification engine, reused as-is, plus a contained composition step
(child-last writer + mark child-owned paths seed). It still delivers the headline use case:
Tolaria packages their full setup, a consumer scaffolds an exact copy — base *and*
overrides — and our base stays refreshable underneath. What it does **not** yet do: give the
*child's* files their own update stream — they're frozen at scaffold time. That is Phase 2.

The rest of Phase 1 is a **refactor**: extract `cli.build`'s hardcoded generators into the
`default` profile behind the generator interface, add profile resolution, and teach the
manifest to record the layer stack and each profile's `source`. `update.run` stops calling
`cli.build` and instead re-resolves the recorded stack.

### 4. Phase 2 — the child gets its own managed, versioned, migratable update stream

The child layer's files become **`managed`**, owned and versioned by the child profile:

- **Per-layer versioning.** Each layer carries its own semver. The manifest records the
  resolved stack as `[{name, version, source, extends}, …]` plus a per-file **owner** index
  into the stack. On update, the engine re-resolves each layer at its latest available
  version and runs `versioning.decide` and `migrations.steps_in_span` **per layer** — these
  are pure functions of `(installed, latest)`, so *they* are reused verbatim — and **gates if
  any layer gates**.
- **Composed migration + regenerate.** Replay each layer's migrations base-first, re-render
  each layer, compose the stack (single-owner rule), then classify. `classify` itself is
  single-manifest/single-seed today, so this is the part that grows: it must be **re-driven
  per owner** (each path compared against its owning layer's old fingerprint and new content),
  not run once over a flat tree. A child that overrode a base path then refreshes that path
  from the *child's* new templates.
- **The release guard is per-author.** We release-guard only the `default`/base profile
  (`tools/release_guard.py`, our cut). Community authors own their own release discipline;
  the engine doesn't police a third party's version labels. The *consumer* is still
  protected — a mistagged child migration still gates the consumer's update downstream
  (`bool(agent_steps)`), the same label-honesty-not-safety-net property we already rely on.

### 5. The profile-author contract: deterministic + resolvable

Two published rules, enforced as preconditions, not hopes:

- **Deterministic.** Same `(profile@version, config)` → byte-identical output. The engine
  can *detect* a violation (a profile whose re-render of a recorded config doesn't match the
  fingerprints it just wrote) and refuse with a clear error rather than silently treating its
  own nondeterminism as a user edit.
- **Resolvable at update time.** `update` must re-find each layer in the environment where
  it runs. A built-in or `contributions/` profile travels with the installed tool. A
  privately distributed profile must be present (a path, an installed package, a fetchable
  pinned source). When a layer can't be resolved, `update` falls back to the existing
  **degraded mode**: add-missing-only for what it *can* resolve, refuse to touch what it
  can't, and report it — the same shape as a missing manifest. This is an honest, surfaced
  limitation, not a silent half-update.

### 6. Trust model: declarative community dir, code profiles opt-in

- The shipped **`contributions/` dir is declarative-only** — data templates, reviewable as
  data in a PR. Rendering is substitution, never code execution.
- **Code profiles** (a Python `generate()` entrypoint via `entry_points`) are a deliberate,
  separately-installed opt-in — *not* accepted into the community dir and never auto-trusted
  from a shared location. They run arbitrary code at scaffold and update time, so they're
  treated like any installed plugin the user chose.
- Even a declarative third-party profile ships content that runs (hooks, MCP servers). So
  scaffolding any **non-built-in** profile names its source, **previews what it will run**
  (§7), and asks for confirmation first — the same "I'm trusting this author" gate as cloning
  and running a template repo, made explicit instead of implicit.

### 7. The command surface: author → consume → update

The architecture above is only as good as the three loops a human (or agent) actually runs,
so each gets a thin, explicit surface. The deep specifics (a `config_schema` language,
richer templating) stay deferred like the rest of §1 — this pins the *workflow*, not every
field.

**Author.** `agent-native-setup profile save <name>` derives a *composing* layer the right
way: it scaffolds `default` into a throwaway tree, **diffs the project against it**, and
captures only the divergent files — new paths and content-overrides, with each file's
seed/managed call — *not* a full copy. (A full copy is all-overrides, which silently severs
base improvements: every path becomes child-owned, §2.) `profile validate <dir>` is the
author-side gate §5 needs *before* publishing: it checks `profile.json` against the schema,
renders, **re-renders and asserts byte-identical** (the determinism contract, caught at
author time instead of on a consumer's `update`), and verifies every `seed` path exists in
`templates/`. The release guard (`tools/release_guard.py`, already a pure `evaluate()`) is
offered as `profile guard <dir> <version>`, opt-in, so an author's version discipline can
match ours.

**Consume.** `--profile <name|path|url>` is a flag on the existing scaffold flow — *not* a
new subcommand (`update` is already the lone subcommand; the default flow stays the
default). `profile list` and `profile show <name>` answer "what's available / what will this
do" (the files it ships, the config it asks, its source) — the discovery the `improvements.md`
draft had, which this RFC must not drop. A `url` form (`git+https://…`) **fetches** the
profile into the user dir, so "distribute within your team" isn't "everyone hand-clones to
the right path." The §6 trust gate **previews what the profile will run** — the SessionStart
hooks and MCP servers it ships, plus its source — before the consumer confirms; a path name
alone can't ground a trust decision. `--trust` (with `--yes`) is the non-interactive escape
hatch, mirroring `update --yes`.

**Update.** The existing loop carries over per layer: the `--check` nudge,
`/update-agent-scaffolding`, `--dry-run`, the gated confirm, the `UPDATING.md` runbook. Two
additions the per-layer world forces:

- A profile declares an **`update_source`** (a releases URL or git ref). `check()` polls
  *our* GitHub today; a privately distributed layer has no feed otherwise, so without this
  the staleness nudge is blind to a new child version. With it, the nudge reports each stream
  independently ("base 0.6→0.7 compatible; tolaria 1.1→1.2 breaking").
- `UPDATING.md` and the gate **label every migration by its owning layer**, and `update`
  applies all resolvable layers in one pass, **gating if any layer gates**. The honest cost: a
  compatible base fix can sit behind a breaking *child* migration in the same run; per-layer
  selectivity (`update --layer base`) is a possible escape hatch, deferred unless the combined
  gate proves painful.

One coupling to state plainly: a `contributions/` profile **ships with the tool**, so "get
`tolaria` 1.2" means upgrading `agent-native-setup` to the version that bundles it — the two
"independent" axes collapse into one for the bundled case. Privately distributed (fetched)
profiles keep the axes independent but lean on `update_source` for the nudge. Both are
honest; neither is hidden.

### User stories

- **Author.** A Tolaria maintainer runs `profile save tolaria` from a representative repo —
  it diffs the repo against `default` and captures only the divergent files (§7) — sets
  `extends: default` and `1.0.0` in `profile.json`, runs `profile validate` until it's clean,
  and either opens a PR to `contributions/` or shares the directory with their team.
- **Consumer, scaffold.** `agent-native-setup --profile tolaria` → previews what it'll run
  and confirms the source, then gets base + Tolaria's house rules, MCP servers, and `.claude/`
  agents in one shot.
- **Consumer, update (Phase 1).** A base fix (sharper CI, new security default) lands;
  `update` refreshes the managed base under their feet; Tolaria's seed files stay put.
- **Consumer, update (Phase 2).** Tolaria ships `1.1.0` (tighter completion checklist);
  `update` re-resolves the `tolaria` layer, gates if the bump is breaking, replays its
  migrations, and refreshes Tolaria-owned managed files — two independent update streams,
  composed.
- **From blank.** A consultant authors `extends: null` for a non-agent-native house style,
  using us purely as an updatable templating engine.

## Consequences

- **The product reframes** from "a wizard" to "an updatable-scaffolding engine with a
  default profile + a profile ecosystem." The wizard becomes one (privileged) profile. This
  is the biggest conceptual shift since the manifest, and it's mostly a generalization of
  code that already exists rather than new machinery.
- **We become a distribution channel.** A `contributions/` dir means review burden,
  namespacing, a quality/security bar, and the trust questions a plugin marketplace carries —
  ongoing cost, not a one-time build. Mitigated by declarative-only contributions (reviewable
  as data) and an explicit trust gate for non-built-in profiles.
- **The workflow surface (§7) is its own build, not free.** `profile save` (diff-against-base),
  `profile validate`/`guard`, `list`/`show`, the `url` fetch, and the trust preview are real
  commands with their own tests and docs — the smoothness of the three loops lives here, so
  it can't be hand-waved as "just the engine plus a flag." Sequenced with the phases: `save`
  + `validate` + `--profile` + the trust preview are Phase 1; richer discovery and the
  `update_source` nudge follow the Phase 2 update stream.
- **Phase 1 ships real value at low engine risk.** Tolaria-style exact-clone scaffolding
  with a still-updatable base reuses the classification/gate/migration engine as-is; the new
  code is the contained composition step (§3 — child-last writer, mark child-owned paths
  seed). The headline capability does not wait on the two-axis update stream.
- **Phase 2 roughly doubles the updater's conceptual surface** — two version axes, per-layer
  gates and migrations, layer-merge in classify. Bounded by the single-owner rule (each path
  has exactly one owning layer), and every primitive it needs (`decide`, `breaking_series`,
  `steps_in_span`, classify) already exists and is reused per layer rather than reinvented.
- **Determinism becomes a published contract, enforced.** Good hygiene anyway, but now a
  profile author can break a *consumer's* update by being nondeterministic — so the engine
  detects and refuses rather than mis-attributing the drift to a user edit.
- **`update` is only as good as resolvability.** A privately distributed profile that isn't
  present at update time degrades to add-missing-only — surfaced, not silently skipped, and
  the same failure shape as today's missing-manifest path.
- **The "untouched semantic break" hole widens for third-party profiles.** RFC 2026-06-22 §8
  already names the one gap the gate can't catch: a behavioural break in a file the user never
  edited, shipped as a non-breaking bump with no agent/manual step — no fingerprint change, no
  `agent_steps`, so no gate. For our base, the release guard pushes against it. §4 deliberately
  does *not* release-guard community authors, so for a third-party layer the hole is both
  present *and* unguarded — a worse position than the first-party base. The honest mitigations
  are the determinism check (an *output* change still surfaces) and the trust gate (§6), not
  the version gate. We accept this rather than pretend the gate covers it.
- **This RFC is the manifest's first external commitment.** The manifest still ships under the
  `RFC-Not-Needed` waiver (greenfield, only us). Adding a layer stack, a per-file owner index,
  and a re-resolvable `source` turns it into a contract third parties depend on — so this RFC,
  not a separate manifest-format RFC, is where that schema gets locked. Called out so the
  lock-in isn't silent.
- **Supersedes the `improvements.md` "config profiles" draft.** That entry's Tier-1
  (serialized `WizardConfig` presets) is subsumed here as a degenerate profile — a child with
  only a `config` and no templates. The config-file contract and the manifest schema are
  settled together, as that draft asked.

## Alternatives considered

- **Replacement-only profiles (no composition).** A profile is always a full standalone
  generator. Simpler — one version axis, no layer-merge — but it forfeits the whole "leverage
  our structure" goal: every author re-ships (and re-maintains) our base, and our base fixes
  never reach them. Rejected as the default; retained as the `extends: null` mode for authors
  who genuinely want a clean slate.
- **Three-way merge across layers.** Merge base + child diffs onto user edits per file.
  Rejected for the same reason RFC 2026-06-20 rejected it for single-layer update: whole-file
  generation plus the single-owner rule makes classify-and-report sufficient, and the agent
  reconciles edits better than a textual merger. Composition adds layers, not a merge engine.
- **One version for the whole stack.** Pretend base and child release together. Rejected as
  dishonest: they have independent cadences, so a single version can't tell a consumer whether
  *their* update is breaking. The two-axis model is the cost of real independence.
- **Code profiles as the primary model.** Maximum power (a profile *is* a Python generator).
  Rejected as the default: arbitrary execution at scaffold and update time is a supply-chain
  surface that a *community* directory can't safely host. Declarative-first keeps the
  contributions channel reviewable; code profiles remain a deliberate, separately-installed
  opt-in.
- **Ship Phase 2 directly (managed child layers from day one).** Rejected as sequencing, not
  direction: the seed-based Phase 1 delivers the headline use case while the two-axis update
  stream is designed and proven, mirroring RFC 2026-06-20's own "land the engine before
  touching the contract" decoupling.
- **A general templating tool (Jinja, rich logic in profiles).** Rejected for now: our
  generators' conditionals live in code, and a declarative profile's needs are mostly
  "ship these files, mark seed vs managed." A templating language is a dependency and a
  sandbox question; revisit if real profiles hit the wall of plain substitution.
